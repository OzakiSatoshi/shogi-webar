# 📋 将棋WebAR アプリ 要件定義書

> 最終更新: 2026-06-25 / 初版
> 開発アプローチ: Unity等のGUIツールを使わず、**Claude Code によるコード生成 × Web AR** のテキストベース開発で完結する。

---

## 1. プロジェクト概要

### 1.1 目的
人間が**物理の将棋盤・物理の駒**で指す対局に対して、スマホのカメラ越しに
**コードで記述したローポリゴンの3Dキャラクター**を物理駒に重ねてAR表示し、
盤上を「キャラクターの戦場」として演出して楽しむWebアプリケーション。

### 1.2 本アプリが「やること / やらないこと」（最重要スコープ定義）

| 区分 | 内容 |
|------|------|
| ✅ やる | 物理駒へのARキャラクター重畳表示（駒ごとの個別マーカー追従） |
| ✅ やる | A-Frameプリミティブのみで作るローポリキャラ（外部3Dモデル不使用） |
| ✅ やる | キャラの**アイドル（待機）アニメーション**の自動再生 |
| ✅ やる | スマホブラウザ単体での即時起動・クロスプラットフォーム動作 |
| ❌ やらない | 将棋のルールエンジン（合法手判定・王手/詰み判定・二歩等の禁じ手） |
| ❌ やらない | 対戦ロジック・ターン管理・AI思考・オンライン通信 |
| ❌ やらない | 移動/撃破/成り等の**能動的なバトル演出**（初版対象外） |
| ❌ やらない | 盤面状態のアプリ内保持・棋譜記録 |

> **位置づけ:** 本アプリは「ゲーム」ではなく、人間の対局に重なる **AR表現レイヤー（ビジュアライザー）** である。
> 将棋を指す・ルールを守るのは人間の責務であり、アプリは見た目の演出のみを担う。

---

## 2. 技術スタック（アーキテクチャ）

| レイヤー | 採用技術 | 補足 |
|----------|----------|------|
| マークアップ/3D空間 | **A-Frame**（HTML5 / WebGL） | プリミティブで3Dキャラを構築・アニメーション制御 |
| ARトラッキング | **MindAR.js（Image Tracking）** | 駒ごとの画像マーカーを認識・追従 |
| 言語 | HTML5 / JavaScript（ES Modules） | ビルドレス構成を基本とする |
| 配信 | 静的ホスティング（HTTPS必須） | カメラAPIにHTTPSが必要。GitHub Pages / Vercel 等 |
| 開発 | Claude Code | コード生成・反復開発 |

### 2.1 なぜ MindAR（画像認識）か（技術判断の記録）
- **iOS Safari は WebXR の `immersive-ar` を未サポート**（2026年現在）。
  WebXR平面認識（Hit Test）を採用すると **実質 Android 専用** になり、クロスプラットフォーム要件を満たせない。
- MindAR.js は **カメラ映像 + WebGL** で動作し、iOS Safari / Android Chrome の双方で動く。
- よって「特別なアプリ不要でiPhoneでも動く」という非機能要件を満たすため、MindAR画像認識を主軸に据える。

## 3. 機能要件

### ① カメラ・AR起動
- スマホブラウザからカメラへのアクセス権限を取得する。
- HTTPS配信下で、ページを開くだけで（インストール不要で）AR表示を開始する。

### ② 駒トラッキング（駒ごと個別マーカー追従）
- 各物理駒に貼付した**画像マーカー**をMindARで認識し、その駒の上にARキャラクターを重畳表示する。
- 人間が物理駒を動かすと、ARキャラクターが物理駒に**追従**する。
- マーカーが画面外/隠れた駒は、対応するARキャラを非表示にする（トラッキングロスト時の挙動）。

#### マーカー設計の方針
将棋の駒は「種類 × 向き」で意味が決まるため、必要なマーカー種類を整理する。

- 基本8種: 歩 / 香 / 桂 / 銀 / 金 / 角 / 飛 / 王(玉)
- 成り6種: と / 成香 / 成桂 / 成銀 / 馬(成角) / 龍(成飛)　※金・王は不成
- → **駒種としては最大14種の意匠**が存在する。

> ⚠️ **本要件で最大の技術的論点（後述「7. 技術リスク」参照）:**
> 同一種類の駒（例: 歩は片側9枚）が盤上に複数存在する。MindARの標準Image Trackingは
> 「同一ターゲット画像の複数インスタンス同時追従」を不得手とする。
> このため初版は **「各駒に固有マーカーを割り当てる方式」** を基本とし、
> 同時追従可能なマーカー枚数を Phase 1 で実測・検証してから本実装範囲を確定する。

### ③ キャラクター表現（ローポリ幾何学記述）
- 外部3Dモデル（.gltf 等）は読み込まず、A-Frameプリミティブ（`<a-box>` `<a-cone>` `<a-cylinder>` `<a-sphere>` 等）の組み合わせで各キャラを表現する。
- 駒種ごとに識別しやすい簡易デザインを与える（下表は初期デザイン案・調整可）。

| 駒 | キャラ・デザイン案 |
|----|----------------------|
| 歩兵 | 赤い直方体の体 ＋ 黄色い円錐の頭 ＋ 槍 |
| 香車 | 細長い直方体 ＋ 前進感のある矢じり頭 |
| 桂馬 | 跳ねる脚を持つ騎乗風シルエット |
| 銀将 | 銀色の鎧（立方体）＋ 剣 |
| 金将 | 金色の鎧（立方体）＋ 重厚な兜 |
| 角行 | 紫の細長い円柱（ローブ）＋ 球体の頭 ＋ 杖 |
| 飛車 | 青い装甲 ＋ 飛翔を思わせる翼パーツ |
| 王将/玉将 | 玉座風の装飾（立方体の組み合わせ）＋ 王冠 |
| 成駒 | 元デザイン ＋ 金色パーツ／オーラ等で「進化」を表現（静的差分） |

> ※ 駒の所有者（先手/後手）は、物理駒の向きに合わせて**キャラの正面向き**で表現する。

### ④ アイドルアニメーション（自動・唯一の動的演出）
- 各キャラは常時、その場で**待機モーション**を自動再生する（例: 上下の呼吸、微小な揺れ、回転、オーラの明滅）。
- A-Frameの `animation` 属性（loop, easing）で実装する。
- **移動・撃破・成り・王手などの能動的演出は初版スコープ外**（将来拡張、9章参照）。

## 4. 非機能要件

- **超軽量・即時起動**: 重い3Dアセットを持たず、一般的なWebページ程度（目安数秒以内）でAR起動。
- **クロスプラットフォーム**: アプリインストール不要で iPhone(Safari) / Android(Chrome) のブラウザで動作。
- **HTTPS必須**: カメラアクセスのため本番・検証ともHTTPS配信。
- **パフォーマンス**: 一般的なミドルレンジスマホで実用的なフレームレートを維持（同時追従マーカー数とのトレードオフはPhase 1で評価）。
- **オフライン親和**: 可能な範囲でCDN依存を明示し、ネットワーク不調時の挙動を定義（任意）。

## 6. 画面・UX（初版）

## 7. 技術リスクと検証事項（重要）

本構成は「駒ごと個別マーカー追従」を採るため、以下を**Phase 1で必ず実測検証**する。

| # | リスク/検証項目 | 内容 | 対応方針 |
|---|------------------|------|----------|
| R1 | 同時追従マーカー数の上限 | MindARが実機で安定追従できる同時マーカー枚数。将棋は最大40駒。 | Phase 1で 1→数枚→多数 と増やし実測。上限に応じて初版スコープ（例: 主要駒のみ）を調整。 |
| R2 | 同一画像の複数インスタンス | 同種駒（歩9枚等）の同時表示。MindAR標準は不得手。 | 各駒に**固有マーカー**を割り当てる方式を基本とする。 |
| R3 | マーカー物理サイズ | 駒は小さく、印刷マーカーの認識限界がある。 | マーカーの特徴量・サイズ・印刷品質を検証。必要なら駒台/駒サイズ前提を定義。 |
| R4 | iOS Safari のカメラ/WebGL制約 | 実機での起動・権限・発熱・性能。 | 早期に実機テスト。 |
| R5 | 追従の安定性・ジッター | 重畳キャラの揺れ・遅延。 | スムージング/トラッキングロスト時の非表示処理で吸収。 |

> R1/R2の結果次第で、初版の現実的な落とし所（例: 「全駒同時」ではなく「画面内の数駒のみ重畳」）を再定義する可能性がある点を、関係者の合意事項とする。

## 8. 開発フェーズ計画

### Phase 0: 環境・土台
- リポジトリ初期化、HTTPS配信（ローカル/本番）確認、A-Frame + MindAR の読み込み確認。

### Phase 1: 技術検証（最優先・本構成の肝）
1. **単一マーカー上に歩兵キャラ1体**を表示（最初の動くプロトタイプ）。
2. アイドルアニメーション追加。
3. マーカーを2枚→数枚→多数と増やし、**同時追従数と性能を実機で実測**（R1/R2）。
4. 実測結果をもとに初版の現実的スコープを確定。

### Phase 2: キャラ拡充
- 駒種ごとのローポリデザインを順次実装（歩→主要駒→全駒）。
- 成駒の静的差分デザイン。

### Phase 3: 仕上げ
- 起動UX/権限フロー、フォールバック、簡易デバッグ表示、配信最適化。

## 🚀 ファーストステップ（Claude Code への最初の指示・案）

Phase 1-1 を以下のプロンプトで開始する。

> 「A-Frame と MindAR.js（Image Tracking）を使い、1枚の画像マーカーを認識したら、その上に
> **HTMLのプリミティブ（a-box / a-cone 等）だけで描いたローポリの『歩兵キャラクター』が1体出現**して、
> その場で上下に揺れる**アイドルアニメーション**を再生する、最小構成の `index.html` を作成してください。
> キャラは赤い箱型の体に黄色いとんがり頭の簡易な見た目で構いません。iPhone Safari / Android Chrome の
> 両方で動くこと、HTTPS配信前提であることを考慮してください。」
